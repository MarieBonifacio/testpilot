### Étape 8.1 — Migration hachage SHA-256 → bcrypt

**Problème :** `hashPassword` utilise SHA-256 sans sel (proxy.js ligne ~73), vulnérable aux attaques par dictionnaire.

**Dépendance à ajouter :** `bcrypt` (ou `bcryptjs` pour pure-JS)
```
npm install bcrypt
npm install --save-dev @types/bcrypt  (si besoin TypeScript)
```

**Fichiers modifiés :**
- `proxy.js` — remplacer `hashPassword()` 
- `routes/auth.js` — adapter `register` et `login`
- `init_db.js` — ajouter colonne `password_version` (migration douce)

**Stratégie de migration douce (sans casser les comptes existants) :**
```js
// Nouvelle fonction (proxy.js)
async function hashPassword(pw) {
  return bcrypt.hash(pw, 12);  // 12 rounds = ~300ms sur hardware moderne
}

async function verifyPassword(pw, hash) {
  // Support de l'ancien format SHA-256 (migration transparente)
  if (hash.length === 64) {  // SHA-256 hex = 64 chars
    const sha = crypto.createHash('sha256').update(pw).digest('hex');
    if (sha !== hash) return false;
    // Ré-hachage automatique avec bcrypt (upgrade silencieux)
    return 'rehash_needed';
  }
  return bcrypt.compare(pw, hash);
}
```

**Logique de login mise à jour (`routes/auth.js`) :**
```js
// POST /api/auth/login
const result = await verifyPassword(password, user.password_hash);
if (!result) return res.status(401).json({ error: 'Identifiants invalides' });

// Migration silencieuse SHA-256 → bcrypt
if (result === 'rehash_needed') {
  const newHash = await hashPassword(password);
  await dbRunP(db, 'UPDATE users SET password_hash = ? WHERE id = ?', [newHash, user.id]);
}
```

**Migration schema (`init_db.js`) :**
```sql
ALTER TABLE users ADD COLUMN password_version INTEGER DEFAULT 1;
```

**Tests Phase 8.1 :** Ajouter dans `tests/auth.test.js` :
- Login avec compte SHA-256 existant → migration automatique vers bcrypt
- Login avec compte bcrypt → fonctionne normalement
- Vérifier que le hash stocké change après migration

### Étape 8.2 — Validation des entrées (Zod ou validation manuelle)

**Problème :** Aucune validation systématique des corps de requête. Un `req.body.role = 'admin'` envoyé par un `automaticien` pourrait passer si les guards ne sont pas exhaustifs.

**Choix technique :** Validation manuelle avec helpers (pas de dépendance externe Zod/Joi pour garder le projet léger).

**Nouveau fichier :** `middleware/validate.js`
```js
// Règles de validation simples
const validators = {
  scenario: (body) => {
    const errs = [];
    if (body.priority && !['high', 'medium', 'low'].includes(body.priority))
      errs.push('priority doit être high/medium/low');
    if (body.scenario_type && !['functional', 'negative', 'boundary', 'performance'].includes(body.scenario_type))
      errs.push('scenario_type invalide');
    return errs;
  },
  user: (body) => {
    const errs = [];
    if (body.role && !['automaticien', 'key_user', 'cp', 'admin'].includes(body.role))
      errs.push('role invalide');
    if (body.username && !/^[a-zA-Z0-9_.-]{3,50}$/.test(body.username))
      errs.push('username invalide (3-50 caractères alphanumérique)');
    return errs;
  },
  apiToken: (body) => { ... }
};

function validate(schema) {
  return (req, res, next) => {
    const errs = validators[schema]?.(req.body) || [];
    if (errs.length) return res.status(400).json({ error: errs.join(', ') });
    next();
  };
}
module.exports = { validate };
```

**Application dans les routes :**
- `PUT /api/scenarios/:id` → `validate('scenario')`
- `POST /api/auth/register` → `validate('user')`
- `PUT /api/users/:id` → `validate('user')`
- `POST /api/user/api-tokens` → `validate('apiToken')`

**Fichiers modifiés :** `routes/auth.js`, `routes/scenarios.js`, `routes/cicd.js`

### Étape 8.3 — Rate limiting

**Problème :** Pas de rate limiting sur `/api/auth/login` (brute force) ni sur `/api/trigger` (abus CI/CD) ni sur le proxy Anthropic.

**Dépendance à ajouter :** `express-rate-limit`
```
npm install express-rate-limit
```

**Configuration (`proxy.js`) :**
```js
const rateLimit = require('express-rate-limit');

// Auth login : 10 tentatives / 15 minutes par IP
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 10,
  message: { error: 'Trop de tentatives de connexion. Réessayez dans 15 minutes.' },
  standardHeaders: true,
  legacyHeaders: false,
});

// CI/CD trigger : 60 déclenchements / heure par token
const triggerLimiter = rateLimit({
  windowMs: 60 * 60 * 1000,
  max: 60,
  keyGenerator: (req) => req.headers.authorization || req.ip,
  message: { error: 'Quota de déclenchements CI/CD atteint (60/h).' },
});

// Proxy LLM Anthropic : 30 req / minute par IP
const llmLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 30,
  message: { error: 'Limite LLM atteinte. Réessayez dans une minute.' },
});
```

**Application :**
```js
// Dans proxy.js, avant les routes :
app.post('/api/auth/login', authLimiter);   // appliqué avant le router auth
app.post('/api/trigger', triggerLimiter);
app.post('/api/messages', llmLimiter);
```

**Fichiers modifiés :** `proxy.js`

**Tests Phase 8.3 :** Ajouter dans `tests/auth.test.js` :
- 11 tentatives login → 12ème retourne 429
- Vérifier header `X-RateLimit-Remaining`

### Étape 8.4 — Audit logging

**Problème :** Seule la table `triggered_executions` trace des événements CI/CD. Les actions sensibles (login, création user, validation scénario, révocation token) ne sont pas tracées.

**Nouveau fichier :** `init_db.js` — ajouter la migration :
```sql
CREATE TABLE IF NOT EXISTS audit_logs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER,
  username TEXT,
  action TEXT NOT NULL,          -- 'login', 'logout', 'create_user', 'delete_user', 
                                 -- 'validate_scenario', 'reject_scenario', 
                                 -- 'create_token', 'revoke_token', 'trigger_session'
  entity_type TEXT,              -- 'scenario', 'user', 'session', 'api_token'
  entity_id TEXT,
  metadata TEXT,                 -- JSON optionnel (ex: ancien rôle → nouveau rôle)
  ip_address TEXT,
  user_agent TEXT,
  created_at TEXT DEFAULT (datetime('now')),
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE SET NULL
);
CREATE INDEX IF NOT EXISTS idx_audit_logs_user ON audit_logs(user_id);
CREATE INDEX IF NOT EXISTS idx_audit_logs_action ON audit_logs(action);
CREATE INDEX IF NOT EXISTS idx_audit_logs_created ON audit_logs(created_at);
```

**Nouveau helper (`middleware/audit.js`) :**
```js
function logAudit(db, { userId, username, action, entityType, entityId, metadata, req }) {
  db.run(
    `INSERT INTO audit_logs (user_id, username, action, entity_type, entity_id, metadata, ip_address, user_agent)
     VALUES (?, ?, ?, ?, ?, ?, ?, ?)`,
    [
      userId || null,
      username || null,
      action,
      entityType || null,
      entityId ? String(entityId) : null,
      metadata ? JSON.stringify(metadata) : null,
      req?.ip || null,
      req?.headers?.['user-agent'] || null,
    ]
  );
}
module.exports = { logAudit };
```

### Étape 8.5 — Sécurisation des tokens API

**Problèmes identifiés :**
1. Les tokens CI/CD n'ont pas de longueur minimum de secret
2. Pas de rotation de token
3. Aucune alerte sur token expiré

**Actions :**

**A) Améliorer `generateApiToken` (proxy.js) :**
```js
// Actuellement : "tpt_" + 32 bytes base64url = ~47 chars
// Conserver tel quel (suffisant), mais ajouter vérification à la création
function generateApiToken() {
  const secret = crypto.randomBytes(40).toString('base64url');  // 54 chars URL-safe
  return `tpt_${secret}`;
}
```

**B) Ajouter endpoint de rotation (`routes/cicd.js`) :**
```js
// POST /api/user/api-tokens/:id/rotate
// Révoque l'ancien token, génère un nouveau avec les mêmes paramètres
router.post('/api/user/api-tokens/:id/rotate', requireAuth, async (req, res) => {
  // 1. Vérifier ownership
  // 2. Générer nouveau token
  // 3. Mettre à jour hash dans la DB
  // 4. Retourner le nouveau token en clair (une seule fois)
});
```

**C) Header de warning sur token expirant bientôt (`routes/cicd.js` dans authMiddleware) :**
```js
// Si token expire dans moins de 7 jours, ajouter header de warning
if (token.expires_at) {
  const daysLeft = (new Date(token.expires_at) - Date.now()) / (1000 * 86400);

}
```
